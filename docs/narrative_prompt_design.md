# Narrative Prompt Design

## The Core Decision

The narrative prompt does exactly three things:

1. State the overall risk picture in plain language
2. Name the specific dimensions driving the risk, citing actual metrics
3. Recommend a concrete next action, sized to the tier

It does NOT:

- Invent metrics not in the input
- Change the tier (the tier is fixed by the rules engine)
- Pad with filler or generic risk language
- Use jargon without grounding it in a specific number

## Why Pre-Compute, Not Real-Time

The most important design decision in this project: narratives are generated by a Python script that runs on a schedule (nightly, or on-demand for dashboard refreshes), and stored as a static column joined into Power BI.

Three reasons:

1. **Latency**: 3-5 seconds per click is unusable in an interactive dashboard.
2. **Cost**: 100 suppliers × concurrent users = unbounded spend.
3. **Reliability**: rate limits during a busy review meeting would break the demo.

A real org refreshing supplier data nightly is the right cadence for risk monitoring. The 24-hour staleness ceiling is acceptable; the latency and cost wins are not.

## The Prompt

See `scripts/generate_narratives.py` for the production version. Key choices:

### System prompt structure

- **Role grounding**: "You are a Supply Chain Risk Analyst."
- **Hard constraints in `MUST NOT` form**: explicit, capitalized.
- **Output schema as JSON**: every field named.
- **Final instruction**: "Return only the JSON. No preamble, no markdown fencing."

### User message structure

The user message is a structured supplier profile, not free text. Format:

```
Supplier: EastBay Magnetics (MFG-073)
Category: Sensors
Region: APAC
Tier: YELLOW (overall score 56.2)

Dimension scores (0-100):
- Quality: 66.0
- Delivery: 42.5
- Capacity: 82.5
- Financial: 60
- Geopolitical: 75.0

Underlying metrics:
- PPAP status: Approved
- On-time delivery: 78.5%
- Quality escapes (60d): 4
- ...

Flagged dimensions: delivery
```

Why this format:

- **Pre-computed dimension scores** so the model doesn't have to do that math
- **Tier already provided** so the model doesn't try to override it
- **Underlying metrics** so the model can cite actual numbers in the narrative
- **Flagged dimensions list** as an explicit hint about what to lead with

## Confidence Flagging

Every narrative includes a `confidence` field (HIGH / MEDIUM / LOW). Why:

- **GREEN suppliers**: confidence is HIGH because there's no real risk to assess
- **RED suppliers**: confidence is HIGH if multiple dimensions clearly flagged
- **YELLOW suppliers**: confidence often MEDIUM because tier-boundary cases need human judgment
- **Suppliers with sparse or conflicting data**: confidence is LOW

This signals to the PM where to spend their human attention.

## Failure Modes Handled

1. **Markdown fencing**: the parser strips ` ```json ` if Claude adds it despite instructions.
2. **Malformed JSON**: falls back to the deterministic mock for that supplier rather than crashing the whole pre-compute job.
3. **No API key**: the entire pre-compute runs on the deterministic mock so the demo works without credentials. Same architecture, deterministic output.
4. **Per-supplier API failure**: caught individually so one bad call doesn't fail the batch.

## What's Different vs. Project 1's Narrative Layer

Project 1 (NPI Readiness Copilot) generates narrative interactively because it's per-gate, not per-supplier-per-day. Different latency tradeoffs. The 3-5 second wait for a once-per-week gate review is acceptable; the same wait for every dashboard click is not.

This is a deliberate architectural distinction that an interviewer will appreciate. AI placement decisions depend on usage frequency, not just on what AI can do.
